01 — Problem Statement

Enterprises with multiple consumer channels (web, mobile, partner APIs, IoT) need a structured, governed approach to expose capabilities without creating point-to-point integration spaghetti. Without a clear API tier, each team builds its own entry point, duplicating concerns like auth, rate limiting, logging, and versioning.

This pattern applies when: (a) you have 3+ consumer types accessing the same backend capabilities, (b) you need consumer-specific contract isolation, (c) you need centralised policy enforcement without coupling it into every service, or (d) you are onboarding external partners needing controlled capability exposure.

02 — Architecture Style
Primary Style: Layered API Architecture

Three-tier API model: Experience APIs (consumer-specific), Process APIs (orchestration), System APIs (data/core systems). Justified because it separates consumption concerns from system implementation — consumers are isolated from backend churn.

Trade-offs

Adds latency per tier (typically 5–15ms per hop). Governance overhead requires an API registry and contract lifecycle tooling. Risk of "API proliferation" if not governed — address via mandatory registration in the API catalogue.

⚡
Scalability
API GW scales automatically. BFF tier on Fargate with horizontal scaling via ECS Service Auto Scaling. Target <200ms p99 at gateway.
🛡️
Resilience
Circuit breakers at process→system tier. Retry with exponential backoff. Graceful degradation returning partial responses.
🔐
Security
Mutual TLS between tiers. JWT with short TTL (15min). API keys for partner tier. OWASP Top 10 mitigated at WAF.
📊
Observability
Distributed tracing via X-Ray. Correlation ID propagated across all three tiers. CloudWatch dashboards per tier.
💰
Cost
Cache at API GW layer (TTL tuned per endpoint). Spot for BFF compute. Lambda for low-traffic system adapters.
07 — Anti-Patterns
✕
God API Gateway
Putting business logic into the gateway (transformations, orchestration). Gateway should only handle cross-cutting concerns — routing, auth, rate limiting.
✕
Bypassing the API Tier
Services calling each other directly, bypassing the gateway layer. Enforced via network segmentation — only the gateway CIDR should be allowed to call BFFs.
✕
Chatty Process APIs
Process APIs making 10–20 downstream calls per request. This signals missing aggregation. Address with GraphQL federation or a dedicated composition layer.
✕
Unversioned APIs
No contract versioning strategy leads to breaking consumers silently. Mandate semantic versioning in the API gateway path (/v1/, /v2/) from day one.
08 — Variants
Variant A
GraphQL Federation
Replace REST BFF with a federated GraphQL gateway (Apollo Federation). Ideal when consumers need flexible querying and over-fetching is a real problem. Higher operational complexity — requires schema governance.
Variant B
Lightweight Internal API Mesh
For internal service-to-service only (no external consumers). Removes CDN/WAF, simplifies auth to IAM SigV4 or SPIFFE/SPIRE. Lower cost, higher throughput. Not suitable for external-facing workloads.
Variant C
Partner API Marketplace
Extends the standard pattern with an API Developer Portal, OAuth2 client credentials per partner, tiered rate limits, and usage billing hooks (AWS Marketplace / custom). Adds API monetisation capability.

PATTERN — 02 / 10
Event-Driven Processing Pattern
EDA Pub-Sub Async Decoupled
01 — Problem Statement

Synchronous, point-to-point integrations create tight coupling: if Service B is slow, Service A blocks. If Service C doesn't exist yet, Service A cannot be deployed. This forces deployments to be coordinated across teams, creating organisational bottlenecks and fragility under load.

Apply when: (a) multiple consumers need to react to the same business event, (b) producers should not know about consumers, (c) workloads have bursty traffic that would overwhelm synchronous calls, (d) cross-domain integrations require audit trails, or (e) eventual consistency is acceptable (SLA > seconds).

02 — Architecture Style
Primary Style: Event-Driven Architecture (EDA)

Services communicate exclusively via events. Producers emit domain events to a broker; consumers subscribe independently. This eliminates temporal coupling — producer and consumer don't need to be alive simultaneously. Enables open/closed principle at the platform level.

Trade-offs

Debugging and tracing is harder — requires distributed tracing end-to-end. Eventual consistency requires consumers to be idempotent. Dead-letter queue management is mandatory, not optional. Schema evolution requires backward/forward compatibility governance.

06 — Non-Functional Considerations
⚡
Scalability
Broker handles burst natively. Competing consumers on SQS scale horizontally. Kinesis shards can be split for throughput.
🛡️
Resilience
DLQ isolation. Idempotency keys on all consumers. At-least-once delivery with deduplication via idempotency table (DynamoDB).
🔐
Security
Encryption at rest (KMS) and in transit (TLS). Resource-based policies on SQS/SNS. Cross-account publishing via EventBridge resource policies.
📊
Observability
Correlation ID in event envelope (mandatory). X-Ray segment per consumer invocation. DLQ depth alarm critical.
💰
Cost
EventBridge for low-volume; SQS for medium; MSK reserved instances for high-volume streams. Avoid per-message overhead on Kinesis for small payloads.
07 — Anti-Patterns
✕
Event as Command
Naming events as commands ("SendEmail") couples consumers to producer intent. Events should describe what happened ("OrderPlaced") — consumers decide what to do.
✕
Fat Events with Full Payload
Including the entire entity in every event creates coupling to a data model and inflates broker storage. Prefer thin events (entity ID + change type) and let consumers fetch if they need full data.
✕
No DLQ Strategy
Deploying queues without DLQs means failed messages are silently discarded after max retries. Every consumer queue in production must have a DLQ with an alarm.
✕
Non-Idempotent Consumers
At-least-once delivery means duplicate events are possible. Consumers that write without checking for prior processing will corrupt state. Idempotency is not optional.
08 — Variants
Variant A
High-Throughput Streaming
Replace EventBridge/SQS with MSK (Kafka). Suitable for >10k events/sec. Adds consumer group management and offset tracking. Higher ops overhead but full Kafka ecosystem (Connect, Streams).
Variant B
Cross-Account Event Bus
EventBridge with cross-account event bus rules. Producer account publishes to its bus; a cross-account rule forwards to consumer account bus. Ideal for multi-account landing zones with domain isolation.
Variant C
CDC-Driven Integration
Remove producer code changes entirely. MSK Connect + Debezium reads the DB transaction log, emits events automatically. Zero producer-side change needed — ideal for legacy modernisation.

PATTERN — 03 / 10
CQRS + Event Sourcing Pattern
CQRS Event Sourcing Microservices Audit
01 — Problem Statement

Systems with high write/read asymmetry struggle when a single data model tries to serve both transactional mutations and complex read queries. An order management system might write 1k orders/sec but serve 50 different read projections (order status, audit history, fulfilment view, finance view) — a single CRUD model cannot scale or evolve to serve all these consumers without degrading write performance or coupling read models to write schema.

Apply when: (a) write and read patterns have significantly different performance characteristics, (b) regulatory compliance requires tamper-proof, replayable audit history, (c) the domain is complex enough that multiple projections of the same data are needed, or (d) temporal queries ("what was the state at time T") are a business requirement.

02 — Architecture Style
CQRS + Event Sourcing

CQRS separates the write model (Command side) from the read model (Query side). Event Sourcing persists state as an append-only sequence of domain events rather than mutable rows. The current state is derived by replaying events. This is the most powerful data pattern for audit-heavy, compliance-driven domains.

Trade-offs

Significant complexity overhead. Read projections are eventually consistent. Event schema evolution requires versioning strategies (upcasters). Not appropriate for simple CRUD domains. Teams require deep DDD knowledge. Start here only if audit/temporal requirements are explicit.

06 — Non-Functional Considerations
⚡
Scalability
Write throughput scales with DynamoDB partitions. Read models scale independently. Projectors fan out to N read stores without affecting write path.
🛡️
Resilience
Event store is immutable — no data loss. Projections can be rebuilt from scratch by replaying the event log. Resilience by design.
🔐
Security
Event store must be write-once, never-delete (WORM). Use DynamoDB with IAM denying DeleteItem on event table. Compliance-grade audit trail.
📊
Observability
Track projection lag (event timestamp vs projection timestamp). Alert on lag >5s in near-real-time projections. Full event replay time is an SRE metric.
💰
Cost
DynamoDB on-demand for event store (write-heavy bursts). Reserve capacity on read stores once baseline is established. Snapshots reduce replay compute cost.
07 — Anti-Patterns
✕
Applying CQRS Everywhere
CQRS adds significant complexity. For simple CRUD services with no audit requirements or read asymmetry, a standard relational model with read replicas is sufficient. Don't over-architect.
✕
Mutable Event Store
The event store is the system of truth — it must never be mutated. Applying UPDATE/DELETE to events destroys the audit trail and breaks projection replay. Enforce at IAM level.
✕
No Snapshot Strategy
An aggregate with 10,000 events requires replaying all 10,000 to derive current state on every write. Without periodic snapshots, write latency grows unboundedly over time.
08 — Variants
Variant A
CQRS without Event Sourcing
Separate write DB (normalised relational) from read DB (denormalised DynamoDB). Simpler to implement. Loses time-travel and full replay capability but suitable for most teams new to CQRS.
Variant B
EventStoreDB / Purpose-Built Store
Replace DynamoDB event store with EventStoreDB or Axon. Native event sourcing features (subscriptions, projections, snapshotting). Higher ops burden but better developer ergonomics.
Variant C
Regulatory Audit Pattern
Extend the base pattern with event encryption at field level (KMS envelope), immutability enforcement (S3 Object Lock for event archive), and compliance reporting projections. Designed for FSBP, GDPR, SOX.
09 — Reusability Guidance
🔧 IaC Modules
  • Module: event-store-dynamodb — table with stream, WORM IAM deny policy, TTL disabled
  • Module: projection-lambda — Kinesis-triggered Lambda with idempotency table
  • Module: snapshot-schedule — EventBridge rule triggering snapshot Lambda per aggregate
📋 Guardrails
  • IAM SCP: Deny DeleteItem/UpdateItem on tables tagged event-store=true
  • All events must carry a schema version field — breaking changes require a new version
  • Mandatory architecture review before adopting this pattern (complexity gate)
PATTERN — 04 / 10
Serverless Compute Pattern
FaaS Serverless Step Functions Event-Triggered
01 — Problem Statement

Many enterprise workloads are sporadic: a document processing job triggered by an S3 upload, a scheduled report, a webhook handler, lightweight API routes, or glue logic between systems. Provisioning long-running containers or VMs for these workloads wastes compute budget and adds operational overhead (patching, scaling, monitoring idle capacity).

Apply when: (a) request patterns are bursty or intermittent (not sustained), (b) function execution time is under 15 minutes, (c) teams lack operational maturity for container fleet management, or (d) you need to optimise cost for long-tail, low-frequency operations.

02 — Architecture Style
Serverless / FaaS

No server management. Compute scales from 0 to thousands of concurrent executions in seconds. Pay per invocation, not idle time. Functions are stateless units of business logic — state is externalised to managed stores (DynamoDB, S3, ElastiCache).

Trade-offs

Cold start latency (mitigated by provisioned concurrency for latency-sensitive paths). 15-minute execution limit. Harder to debug locally. Not suited for long-running, CPU-intensive, or memory-heavy workloads (>10GB). Vendor coupling is real — abstract with SAM/CDK.

Choreography (EDA)S3 events → Lambda → downstream; no orchestrator needed for simple linear flows
Saga (Step Functions)For multi-step workflows with compensating transactions — externalises orchestration state
Fan-out / Fan-inParallel Lambda invocations via Step Functions Map state; results aggregated post-parallel
Strangler FigIncrementally replace legacy batch jobs with event-triggered Lambda equivalents

06 — Non-Functional Considerations
⚡
Scalability
Lambda scales to 1000 concurrent by default (request limit increase available). Reserved concurrency caps per function to prevent noisy-neighbour throttling.
🛡️
Resilience
Lambda automatically retries async invocations twice. Step Functions handles retries and catch blocks. DLQ for all async paths.
🔐
Security
Least-privilege Lambda execution role. No Lambda in public subnets. VPC endpoints for private service access. Secrets via AWS Secrets Manager (never env vars).
📊
Observability
Lambda Powertools (structured logging, tracing, metrics) is mandatory for all production functions. CloudWatch Lambda Insights for memory/duration anomalies.
💰
Cost
ARM64/Graviton saves ~20%. Right-size memory (use Lambda Power Tuning tool). Avoid large VPC Lambda deployments for cold-start-sensitive paths.
07 — Anti-Patterns
✕
Stateful Lambda
Relying on in-memory state across invocations. Lambda instances may be reused or recycled unpredictably. All state must be externalised — DynamoDB, ElastiCache, S3.
✕
Lambda Monolith
A single Lambda handling 20 routes via an internal router. Defeats the purpose. Leads to large deployment packages, slow cold starts, and impossible fine-grained scaling. One function = one responsibility.
✕
Synchronous Chain of Lambdas
Lambda A calls Lambda B synchronously which calls Lambda C. This creates latency compounding and brittle chains. Use Step Functions for orchestration or SQS for async chaining.
08 — Variants & Reusability
Variant A
Serverless API (HTTP)
API GW (HTTP API) + Lambda + DynamoDB. Minimal API for CRUD operations. Use AWS SAM or CDK constructs. Sub-day deployment to production for new APIs.
Variant B
Serverless Workflow
Step Functions (Standard) + multiple Lambda states. For multi-step business processes with compensation logic. Replace legacy job schedulers gradually with this pattern.
Variant C
Serverless Event Processor
SQS/Kinesis → Lambda. For bulk async processing. Tune batch size and concurrency per workload. Idempotency table in DynamoDB mandatory.
🔧 IaC Modules
  • Module: lambda-standard — function + IAM role + X-Ray + Powertools layer + DLQ
  • Module: step-functions-workflow — state machine + CloudWatch log group + X-Ray
📋 Guardrails

  • No hardcoded secrets in function code — Secrets Manager integration enforced
  • Reserved concurrency must be set on all production functions

Data Lakehouse Pattern
Analytics Delta Lake Lambda Architecture ML Ready
01 — Problem Statement

Enterprises accumulate data across dozens of systems — ERP, CRM, IoT, transactional databases — and need a unified platform for analytics, BI reporting, and ML feature engineering. Traditional data warehouses are expensive and rigid; traditional data lakes lack ACID transactions and fine-grained access control. Neither alone satisfies the full spectrum of analytical requirements.

Apply when: (a) raw data from multiple systems needs to coexist with curated analytical datasets, (b) self-service analytics for business teams is required, (c) ML teams need a governed feature store, or (d) data retention and compliance require tiered storage with audit controls.

PATTERN — 06 / 10
Saga Orchestration Pattern
Distributed Tx Orchestration Compensation Long-Running
01 — Problem Statement

Microservices architectures prohibit distributed two-phase commits (2PC) — they are slow, blocking, and create cross-service coupling at the database level. Yet business transactions often span multiple services: an order flow touches inventory reservation, payment processing, and fulfilment scheduling. If payment succeeds but fulfilment fails, the system must compensate — automatically.

Apply when: (a) a business transaction spans 3+ microservices with distinct data stores, (b) each step must either succeed or be explicitly rolled back via compensating action, (c) transactions can be long-running (seconds to hours), or (d) you need full audit visibility over distributed workflow state.

04 — NFRs & Anti-Patterns
🛡️
Resilience
Step Functions retries with backoff per state. Compensations are guaranteed to run on failure. State is durable — survives Lambda restarts.
🔐
Idempotency
Critical. Each participant must check idempotency key (sagaId + stepId) before acting. Prevents double-charge, double-fulfilment on retry.
📊
Observability
Step Functions execution history in CloudWatch. Custom EventBridge events at saga start/complete/fail. X-Ray across all Lambda participants.
⚡
Scalability
Step Functions supports thousands of concurrent executions. Participant services scale independently. No shared lock between services.
💰
Cost
Step Functions Standard charged per state transition. For <5min high-volume sagas, use Express Workflows — ~2000x cheaper per execution.
✕
Saga Without Compensation
Defining a saga without explicit compensating actions for every forward step. If payment succeeds and fulfilment fails, you need a refund path — always design compensation upfront.
✕
Non-Idempotent Participants
Step Functions will retry failed states. If the payment service charges twice on retry without idempotency checking, the saga is broken. Idempotency is participant-side, not orchestrator-side.
05 — Variants & Reusability
Variant A
Choreography-Based Saga
No central orchestrator. Each service listens for events and emits success/failure events. More decoupled but harder to debug. Prefer for simple 2-3 step flows.
Variant B
Human-in-the-Loop Saga
Saga pauses at a decision step awaiting human approval (TaskToken pattern). Use for high-value orders, compliance gates, or exception handling requiring manual review.
Variant C
Express Workflow Saga
For high-volume, short-duration sagas (<5min). Step Functions Express. No execution history persistence — log all state transitions to CloudWatch explicitly.

01 — Problem Statement

Traditional perimeter security assumes that traffic inside a VPC or data centre is trusted. ⚡
Scalability
Partition input data for parallel processing. Glue DPU autoscaling. AWS Batch scales from 0 to 1000s of containers. Design for data parallelism from day one.
🛡️
Resilience
Idempotent jobs with checkpointing. Failed jobs auto-retry with backoff. Step Functions compensates on terminal failure. Dead-letter output bucket for bad records.
🔐
Security
No long-lived credentials in job definitions. ECS Task Roles / IAM for Batch containers. S3 output encrypted (KMS). Sensitive data masked in CloudWatch logs.
📊
Observability
Job duration tracked in CloudWatch. Record count emitted as custom metric. Missed schedule window alarm critical for financial batch jobs.
💰
Cost
EC2 Spot for Batch (60–80% saving). Glue workers scaled to job size. EMR Serverless scales to 0 between jobs. Schedule outside peak hours to avoid on-demand contention.
✕
Non-Idempotent Batch Jobs
A batch job that cannot be safely re-run after failure will corrupt data on retry. All batch jobs must be designed to be restartable from a checkpoint without duplicating output.
✕
Treating Batch as Streaming
Running batch jobs every minute to simulate "near real-time." If the business need is sub-minute, use the Event-Driven pattern instead. Batch overhead per invocation is non-trivial.
✕
No Missed-Schedule Alert
A batch job that silently fails to trigger (e.g. Scheduler bug, Lambda throttle) will go undetected without a heartbeat alarm. Every scheduled batch job needs a "did it start?" alarm.
